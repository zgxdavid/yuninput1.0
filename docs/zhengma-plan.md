# 郑码接入说明

## 当前判断

- 工程现阶段已经具备 TSF 外壳、候选窗、词频记忆和基础码表查询能力。
- 工程此前缺的不是“中文显示”本身，而是“真正的郑码基础表”和“可复用的导入链路”。
- Linux 生态里可以找到开源的郑码表，至少已经确认 Fcitx 额外码表仓库包含以下表名：
  - `zhengma`
  - `zhengma-large`
  - `zhengma-pinyin`

## 现阶段采用的工程策略

1. 不在仓库里硬编码一份来源不明的完整郑码大词库。
2. 先把 yuninput 做成“可导入开源郑码表”的状态。
3. 安装时复制 `data/` 目录下全部 `.dict` 文件，而不是只复制单个样例表。
4. 候选排序优先精确码，再看用户频率、静态频率、码长和原始顺序，更接近形码输入法。

## 郑码逻辑概述

- 郑码属于形码输入法，核心是按字形拆分，而不是按读音拼写。
- 字根编码是双码体系，实际取码时会根据字根数量和结构决定取哪些码位。
- 对单字来说，完整编码一般比前缀码更优先。
- 对词组来说，通常还需要“构词码”或类似规则来由单字编码生成词编码。

当前 yuninput 还没有实现完整的郑码拆根器。
现阶段接入方式仍然仍以“表驱动”为主：直接加载外部现成码表。

2026-03-23 起，自动造词路径已接入第一版构词码规则：

- 二字词：取首字前两码 + 次字前两码
- 三字词：取第一字首码 + 第二字前两码 + 第三字首码
- 四字及以上：取前三字首码 + 末字首码

这还不是完整郑码规则层，但至少把自动造词从“直接拼接上屏编码”推进到了“按郑码词码规则生成四码词码”。

同日，导入链路已开始保留 Fcitx 码表头中的规则元数据：

- 可写入 `.dict` 文件头部注释
- 也可写入同名 `.dict.rules` sidecar

运行时会优先读取这些规则定义，而不是只依赖代码内置回退规则。

## 导入格式

yuninput 当前支持的 `.dict` 目标格式是 UTF-8 文本，每行一条：

```text
编码 文字
编码 文字 频率
```

例如：

```text
zg 中国
rm 人民
```

## 导入脚本

新增脚本：

- `scripts/import_table_dict.ps1`

用途：把 Fcitx / IBus Table 一类文本码表导入为 yuninput 可直接加载的 `.dict` 文件。

示例：

```powershell
Set-ExecutionPolicy -Scope Process Bypass
./scripts/import_table_dict.ps1 -SourcePath C:\temp\zhengma-large.txt -OutputPath ./data/zhengma-large.dict
```

导入完成后，重新运行安装脚本即可把所有 `.dict` 一并部署到：

- `%LOCALAPPDATA%\yuninput\data`

## 下一步

1. 找到一个许可清晰、可分发的开源郑码基础表或大字集表。
2. 用导入脚本转成 `data/*.dict`。
3. 重装并验证常用单字、词组和重码顺序。
4. 如果要接近冰凌郑码，再补：
   - 构词码规则
   - 右下角语言栏菜单
   - 配置界面与语言栏集成
