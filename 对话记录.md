## 2026-04-18 最新补记

这一天的重点不是继续扩功能，而是把系统兼容码和真实学习链路的边界重新收紧，并准备新的长测版本。

这轮确认的核心判断是：

- `yuninput_user-extend.dict` 本质上是 `zhengma-all.dict` 中两字以上词组的兼容编码表，不应该在首轮排序里和主词库正式词条抢前排。
- 真正应该优先的仍然是系统 GB2312 单字、两码三码字词，以及 `zhengma-all.dict` 中正常的正式词组。
- 非 GB2312 单字仍然不参与调频，`zhengma-all.dict` 中五码词组也不参与调频。
- 后续学习更应该看“汉字词本身”，而不是把“兼容编码 + 汉字词”作为一个越学越靠前的组合条目。

因此本轮收口成三件事：

1. 把系统兼容编码来源和真正 auto phrase 来源拆开，不再复用同一套标记。
2. 在引擎首轮排序和最终展示排序中，把系统兼容码整体后移，只保留在非 GB2312 单字之前。
3. 提交学习时，兼容码只累计词本身的词频，不再累计兼容编码频次。

配套还做了两项工程化动作：

- 排序探针补了兼容词典加载和来源标记输出，方便直接验证 `system` 与 `compat` 的相对顺序。
- 因为仓库位于 Y 盘网盘，验证时统一把 probe 和词典暂存到本机临时目录执行，避免把网盘拒绝运行误判成程序失败。

本轮最后的交付目标明确为：

- 更新对话记录、真实对话、说明书和版本元数据。
- 全量重新编译并在仓库根目录生成 `Yuninput1.3.3.msi`。
- 仍由用户手工卸载旧版本、重启、安装和测试。

## 一、项目起点

项目目标逐渐收敛为：在 Windows 11 上做一个可持续迭代的 TSF 中文输入法工程，而不是只做一个能跑的原型。

从现有仓库和文档能恢复出的核心目标包括：

- 做一个以郑码为核心的 Windows 输入法。
- 不只要“能输入”，还要有候选窗、排序、学习、自动造词、配置工具和安装包。
- 要能编译、打包、安装、清理、诊断和长期回归。
- 要能持续接受真实使用反馈，然后快速收敛问题。

项目早期就不是单点功能开发，而是整套工程化建设：

- C++ TSF Text Service
- Win32 候选窗
- PowerShell 构建、安装、卸载、诊断脚本
- C# 配置工具
- WiX MSI 打包链路
- 默认词库、用户词典、规则文件和导入脚本

## 二、这次 AI 协作里最重要的共识

随着多轮往返，协作方式逐渐固定下来，后续效率明显更高。最关键的几条是：

1. 问题优先按“体感”和“实际使用场景”描述，不先陷入实现细节。
2. 每次代码改动后，都执行完整交付流程，而不是只看编译通过。
3. 代理不直接替你安装最终版本，只负责交付可测试的 MSI。
4. 安装包默认输出到仓库根目录，避免散落到父目录造成混乱。
5. 真正要解决的是根因，不是让某个现象暂时看起来没了。

后来固定下来的工作流是：

1. 清理当前机器上的旧安装态。
2. 全量重新编译 Release。
3. 重新构建相关工具。
4. 重新打包 MSI 到仓库根目录。
5. 由你手动安装和长时间测试。

这套流程虽然慢一点，但明显减少了“以为修好了，实际上测试的不是最新版本”的风险。

### 4. 分页键与模式切换阶段

2026-03-30 的交接里保留了一个很典型的问题：

- 全码提示乱码基本解决。
- 但页键行为还没解决，尤其是英文模式切回中文模式后的 `-`、`+` 相关翻页链路。

这一类问题的价值在于，它提醒出一个很重要的协作经验：

- 不要只说“翻页不对”，要给出最小复现路径。

当时建议的复现步骤就非常具体：

1. 切到英文模式。
2. 再切回中文模式。
3. 触发候选列表。
4. 验证全角减号、加号、小键盘按键的行为。

这类“带上下文的复现”比笼统说“分页有问题”要高效得多。后面的很多排查，都是靠这种明确复现方式才收敛的。

### 5. 安装后 Win+Space 卡死阶段

2026-04-07 是一轮很关键的系统级稳定性修复。

当时用户侧问题是：

- 安装后用 Win+Space 切换输入法时，系统像被拖挂，严重时需要重启或重新登录。

最后归纳出的高概率根因很清晰：

- `Activate()` 热路径里做了太多重活，包括同步字典加载、用户数据迁移、工作线程启动、候选窗创建。
- `Deactivate()` 里又做了阻塞式线程停止和 `join()`。

也就是说，输入法切换这个系统热路径里混进了大量不该同步做的初始化和清理逻辑。

本轮采取的方向是：

- 运行时初始化改成按需懒加载。
- 候选窗改成按需创建。
- 日志线程的生命周期移出每次切换路径。
- `Activate()` 和 `Deactivate()` 只保留必要动作。

这一轮说明一个非常重要的工程经验：

- 在系统级组件里，能不能切换、能不能激活，往往比“首次进来顺手把所有东西都准备好”更重要。

换句话说，热路径要尽可能轻，慢活要延后，重活要懒做。

### 6. 自动造词闭环修复阶段

2026-04-08 上午的重点是自动造词和 helper 链路。

用户感知层面的问题大致是：

- 自动造词不是完全正常。
- 候选窗里一些提示文案也显得奇怪。

后来逐步确认：

- helper 本体不是坏的，手动烟测能正常产出大量 session entry。
- 真正的问题更像是主进程没有可靠跟踪 helper 生命周期。
- helper 完成后，主进程也没有稳定回收结果并刷新候选。

因此，真正有效的修复不是“重写算法”，而是把主进程和 helper 的协作关系补完整：

- 增加 helper 启动、失败、退出日志。
- 保存子进程句柄，避免重复并发拉起。
- 在刷新候选时回收已结束 helper，并强制重载 session 结果。

同时还顺手修掉了一个 UI 层面的误导：

- 原先固定显示“吞码 N”，后来收敛成只有存在剩余编码时才显示“余码 N”。

这里的经验很明确：

- 用户说“功能不正常”，有时不是算法错，而是生命周期、进程协作和结果回收没闭环。
- UI 文案本身也会制造“功能看起来坏了”的错觉。

### 7. 首屏候选速度优化阶段

2026-04-08 下午开始，主要目标转成候选首屏体感。

核心决策是：

- 输入编码时先保证第一页候选尽快出来。
- 后续页、完整扩展候选不必强行塞进同一击键路径。

这背后的取舍非常合理：

- 用户最在意的是“字打下去后，候选是否马上出来”。
- 用户不一定要求每次击键时，后面几页候选也都瞬间准备完毕。

因此，这阶段逐步形成了“首屏优先”的思路：

- 首屏走快路径。
- 主动翻页时再补全扩展候选。
- helper 词库刷新也走轻量路径。
- 继续保留预创建和缓存，降低首次弹窗冷启动。

这是一次很典型的体验型优化：

- 它不一定让系统所有指标都最低，但它显著改善了用户主观响应速度。

### 8. 两阶段候选刷新阶段

同一天稍晚，首屏优化进一步演变成“两阶段候选更新”：

- `RefreshCandidates(false)` 先出首屏同步结果。
- 如有必要，再由内部定时器延后触发 `RefreshCandidates(true)` 去补全扩展候选。

这个方案比“一次性全做完”更平衡：

- 保住首屏体感。
- 不引入额外 UI 线程，避免 TSF/UI 跨线程风险。
- 需要时还能自然补齐更多候选。

同时，这一轮还保留并延续了前面的有效优化：

- 增量前缀缓存
- Backspace 快速恢复
- 轻量测宽
- anchor 复用
- 延迟日志

这说明到这时候，项目已经进入“不是单点修 bug，而是不断叠加局部有效策略”的状态。

### 9. 候选窗抖动与闪烁阶段

2026-04-10 的这轮工作，核心是候选窗稳定性。

你的反馈已经比较明确：

- 当前版本整体还可以。
- 窗口抖动已经明显减轻。

这轮落地的关键动作是：

- `InvalidateRect(..., TRUE)` 改为 `FALSE`，避免背景反复擦除。

### 10. 1.2.1 到 1.2.6：继续压热路径抖动与首屏迟滞

在 v1.2 保存之后，项目并没有停在“功能完整”这一层，而是继续围绕真实输入体感做小步快跑式修整。

因此后续 1.2.1 到 1.2.6 的连续修正，基本都集中在这些方向：

- 模式切换和激活路径继续减重，避免冷启动时把不必要工作塞回热路径。
- 首拍、首屏、自动上屏后的候选恢复继续做快路径和延迟补全。
- 用户数据刷盘进一步后移、批量化，避免在连续输入中制造周期性小卡顿。
- 候选排序和短前缀查询继续收窄边界，避免“看起来像卡住”的空白期。

这一段的价值不在于某一个大功能，而在于：

也正是这一阶段，让你开始明确区分两种问题：

后者虽然不如前者显眼，但在输入法项目里同样重要。

### 11. 配置工具从“能打开”进入“能长期使用”阶段

随着主输入链路逐步稳定，配置工具本身的问题开始集中暴露出来：

- 窗口初始高度不对，底部按钮会被遮住。
- 窗口最大化后，控件没有跟着重排，挤在左上角。
- ESC 退出没有真正生效。
- “打开说明书”想走系统默认浏览器，但实际报错。
- 用户词典管理窗口虽然有入口，但打开像是空的，也没有真正把文件内容加载进来。
- “造词审阅模板”和“管理用户词典”之间职责重叠，入口显得混乱。

这一轮的处理重点，不再是“再加一个按钮”，而是把配置工具重新整理成真正可长期使用的管理界面：

- 主窗口改成可正常拉伸，底部按钮区不再被裁掉。
- 子窗口统一接入 Esc 关闭。
- “打开说明书”改成直接用 Notepad 打开安装目录里的本地说明书。
- “管理用户词典”改成真正同时加载手工用户词条和自动造词，而不是只盯着一个文件或一个标签视图。
- “造词审阅”改成更偏日志和审阅用途，避免和真正的词典管理重复承担编辑职责。

这轮说明一个很现实的经验：

- 工具界面只要开始承载真实管理工作，就不能再停留在“临时小面板”的水平。

### 12. 四码上屏偶发轻微迟滞的根因继续下探

你对 1.2.6 的反馈很典型：

- 整体已经快了一些。
- 四码上屏偶尔还有一点点迟滞。
- 这点迟滞已经能忍，但仍值得再看源代码有没有剩余优化空间。

这类反馈非常有价值，因为它不是“彻底卡住”，而是指出一种还残留在输入体感里的周期性小顿挫。

继续检查后，确认出一个值得修的后台热点：

- 之前自动造词相关的脏标记和手工用户词典写回混在一起。
- 结果是某些自动造词刷盘周期里，会顺手把整份手工用户词典也重新序列化一遍。

它不会每击键都触发，所以不容易在代码表面第一眼看出来；
但一旦触发，就很容易表现成你说的那种：

- 偶尔有一下轻微迟滞。

因此后续处理方向是：

- 把手工用户词典和自动造词运行时词典的脏标记拆开。
- 自动造词刷盘时，不再顺手重建整份手工词典快照。
- 用户数据文件戳监视目标也一并纠正，避免热重载和外部编辑判断盯错文件。

这是一次很典型的“继续修根因，而不是做表面缓解”的优化。

## 四、中间那些无效、打断和返工，真正说明了什么

这部分最值得保留，因为它直接决定下次怎么更快沟通。

### 1. 只看“代码改了”不够，必须看“实际交付链路”

项目里多次出现这样的问题：

- 编译没问题。
- 代码逻辑也看起来没问题。
- 但安装后的行为仍不对。

原因往往不是代码本体，而是：

- 没重新清理旧安装。
- 测的不是最新包。
- MSI 少带了某个关键产物。
- 用户环境里还有残留注册信息。

这也是为什么后来必须固定成完整流程，而不是“我改完你直接试”。

### 2. 自动造词问题，很多时候不是算法错，而是链路没闭环

这一块是很典型的返工来源。

表面看像：

- 自动造词不稳定。
- 候选里看不到预期结果。

但真正的根因可以落在完全不同的层面：

- helper 没被可靠拉起。
- helper 结果没回收。
- MSI 打包遗漏 helper 可执行文件。
- 字典 sidecar 规则没有一并加载，导致构码退化。

这类问题提醒出一条很重要的经验：

- 对“链路型功能”，必须按数据流逐段验证，不能只盯最后显示层。

### 3. 热路径上的重操作，是系统级卡顿的主要风险源

Win+Space 卡死问题基本把这个结论坐实了：

- 同步 IO
- 线程启动和停止
- 候选窗创建
- 字典加载和迁移

这些东西放在系统切换热路径里，迟早会出问题。

所以下次只要再遇到“切换卡、首次进来慢、输入法像拖挂系统”这类问题，第一优先级就该检查：

- 有没有重活跑进 `Activate()`、`Deactivate()`、按键主路径或 UI 绘制路径。

### 4. 体感优化要抓首屏，不要试图一口气做完全部工作

候选系统是这次最典型的例子。

如果目标是用户主观感觉更快，那么正确问题往往不是：

- 为什么所有候选没一起准备完？

而是：

- 为什么第一页没有先出来？

这也是首屏快路径和两阶段刷新最终成立的原因。

### 5. UI 抖动、闪烁和轻微回摆，虽然不是“功能坏了”，但会显著降低产品感

你后面的反馈已经证明了这一点。

即使核心功能已经可用，如果：

- 窗口轻微抖
- 宽度来回跳
- 重绘闪白

用户也会觉得“不稳”。

因此这些问题不该被当成纯 cosmetic，而应视作体验层的真实缺陷。

## 五、这次协作中最有价值的沟通方式

从结果看，下面几种表达最能提高效率。

### 1. 直接描述体感

高效例子：

- 候选窗还是有点抖。
- 首屏出来慢。
- Win+Space 会卡住。
- 自动造词不完全正常。

这类描述虽然不是技术语言，但非常有用，因为它指向真实使用问题。

### 2. 给出具体操作路径

更高效的描述方式是：

- 切到英文，再切回中文，然后按减号翻页不生效。
- 安装后第一次切输入法容易卡。
- 连续输入时第一页候选出得慢，但翻页后后面候选又很多。

这能直接把排查范围缩小到事件路径，而不是整个系统。

### 3. 明确交付规则

后期效率提高，很大程度上是因为你把规则说清楚了：

- 每次改完都要全量编译。
- 每次都先清理旧安装。
- 不要由代理直接安装。
- 只交付仓库根目录的 MSI。

这些规则一旦固定，后面很多歧义就消失了。

## 六、如果下次继续协作，最推荐你怎么提需求

建议使用下面这种模板。它不需要很正式，但信息要完整。

### 模板 A：报问题

1. 现象：发生了什么。
2. 场景：在哪个应用、什么输入步骤下出现。
3. 频率：必现、偶现、长时间后出现。
4. 影响：只是难看，还是会卡死、误上屏、不能翻页。
5. 期望：你更希望先修体感、先修稳定性，还是先保功能。

示例：

- 现象：候选窗还有轻微抖动。
- 场景：连续输入并快速删改编码时更明显。
- 频率：高频出现，但不至于卡死。
- 影响：功能能用，但体感不稳。
- 期望：先优先稳住窗口，不急着再加新功能。

### 模板 B：要求交付版本

1. 目标：这次要解决什么。
2. 边界：不要动什么。
3. 验证方式：你会怎么测。
4. 交付要求：是否需要编译、清理、打包、是否上传。

示例：

- 目标：先修候选窗闪烁和轻微跳动。
- 边界：不要改排序逻辑。
- 验证方式：我手动安装后长时间打字测试。
- 交付要求：完整重编译，生成根目录 MSI，不安装，不上传。

### 模板 C：做复盘

1. 这次版本主观感受如何。
2. 比上一版更好的是哪几项。
3. 还没解决的核心问题是什么。
4. 下一轮优先级怎么排。

## 八、给这次第一次 AI 做应用的结论

如果把这次过程压缩成几句话，比较准确的结论是：

- 你不是在“让 AI 一次性生成一个成品”，而是在和 AI 一起做连续迭代。
- 真正高效的不是一次把需求说满，而是先说体感和问题，再逐步补复现路径和交付约束。
- AI 最适合做的是：快速读代码、持续收敛根因、批量修改、重复执行构建和发布流程、把零散结果整理成结构化文档。
- AI 最不该做的是：在验证不充分时假定问题已经彻底解决。

这次最成功的地方，不是某一个 commit，而是你把协作规则逐渐定清楚了。规则一旦明确，后面的推进速度明显更快，返工也更少。
